# Architecture

## Next.js 16.2.6 conventions that shape this app

Confirmed from `node_modules/next/dist/docs` (this version differs from older
Next.js — these are load-bearing):

- **App Router, Server Components by default.** Data fetching happens in async
  Server Components; interactivity is isolated to `'use client'` leaf
  components.
- **`params` and `searchParams` are Promises.** Pages must
  `await params` / `await searchParams`. Use the global `PageProps<'/route'>` /
  `LayoutProps<'/route'>` helper types (generated by `next dev`/`build`).
- **Mutations are Server Functions** (`'use server'`), invoked via `<form
  action={fn}>`, `formAction`, or from client event handlers. Always
  re-validate authorization (here: owner/participant token) **inside every
  Server Action** — they are reachable by direct POST.
- **Cache Components is enabled** (`cacheComponents: true` in
  `next.config.ts`). Consequences:
  - Anything that reads uncached data or runtime APIs (`cookies()`,
    `headers()`, `params`, `searchParams`, DB reads) **must** be inside a
    `<Suspense>` boundary or marked `'use cache'`, or the build/dev errors with
    *"Uncached data was accessed outside of `<Suspense>`"*.
  - Cache with the `'use cache'` directive + `cacheLife(profile)` +
    `cacheTag(tag)`. Invalidate with `updateTag` (Server Actions only,
    immediate, read-your-own-writes) or `revalidateTag` (Server Actions
    **and Route Handlers**, stale-while-revalidate). `refresh()` from
    `next/cache` refreshes the client router without touching tagged caches.
  - **Non-deterministic ops** (`crypto.randomUUID()`, `Date.now()`,
    `Math.random()`) are **forbidden in a `'use cache'` or
    prerendered scope** — they must be deferred to request time. In this app
    all ID/token generation happens **only inside Server Actions**
    (request-time, allowed); a WS-8.3 audit asserts `lib/ids.ts` and these
    ops are unreachable from any cached scope (resolves review B5).
- **`unstable_instant`**: export `const unstable_instant = { prefetch:
  'static' }` from routes that must navigate instantly. It validates at
  dev/build that every entry point (page load *and* sibling client nav under a
  shared layout) has a correct static shell. Routes that are inherently
  per-user/dynamic opt out with `export const unstable_instant = false` on
  their layout.
- **Route Handlers** live in `route.ts`; cannot coexist with `page.tsx` at the
  same segment. Use the `RouteContext<'/path'>` helper for typed params.
- `redirect()` from `next/navigation`; `<Link>` from `next/link`.

### Caching policy for this app

| Data | Strategy |
|---|---|
| Campsite catalog & detail | `'use cache'` + `cacheTag('campsites')` + **`cacheLife('days')`** (pinned, incl. `/campsites/[id]`). Effectively static; re-imported rarely. |
| Campsite search results | Cached function keyed by normalized query args; `cacheTag('campsites')` + `cacheLife('hours')`. |
| Catalog invalidation | Importer/reseed pings dev-only `app/api/revalidate-campsites/route.ts` which calls `revalidateTag('campsites')` (Route Handlers may; scripts run outside the request lifecycle so cannot call it directly). v1 limitation: otherwise a restart refreshes the catalog. (Resolves review I-A.) |
| Trip (items, participants, claims) | **Never cached.** Rendered per request under `<Suspense>`. Mutations call `updateTag('trip:<id>')`. |
| Packing template + amenity rules | Pure data/computation — runs in the static shell, no cache needed. |

### Instant-navigation discipline

- `/` , `/campsites`, `/campsites/[id]` export
  `unstable_instant = { prefetch: 'static' }`. Campsite content is cached;
  search results and any runtime reads stream behind their own `<Suspense>`
  fallbacks. Resolve `params` inline (`params.then(...)`) when passing into a
  `'use cache'` child so the cached component gets a plain value.
- `app/trips/layout.tsx` exports `unstable_instant = false` — trip pages are
  per-visitor and uncached by design; we don't promise an instant static shell
  there, but the rest of the app stays validated.

## Data flow

### Browse → select campsite
`/campsites?q=...` → `searchCampsites(args)` (`'use cache'`, tagged
`campsites`) → list of `CampsiteCard` → `/campsites/[id]` cached detail +
`AmenityGrid`.

### Create a trip
`StylePicker` form on the campsite page submits `createTrip(formData)`:
1. Validate `{ campsiteId, style }` with zod.
2. Generate the packing list: `generate(style, campsite.amenities)` →
   categorized items with `scope` flags.
3. Insert `trip` (new UUID slug) + `trip_items`. **Auto-join the creator as
   participant #1** (`is_owner=1`). Set **both** cookies: `bc_owner` and
   `bc_participant` (httpOnly, sameSite=lax, path `/trips/<id>`, ≥128-bit
   tokens). The creator is therefore owner *and* a participant — so the
   solo-trip `participantCount` is 1 and per-person `requiredQty` is correct
   (resolves review B6).
4. `redirect('/trips/<id>')`.

The creator "selecting which items they'll bring" before sharing is just
that participant placing ordinary claims — they persist into the shared trip
and show under "who's bringing what" (design decision DR-4, review G2).

### Edit list (owner)
`addItem` / `updateItem` / `removeItem` / `reorderItem` / `renameTrip` Server
Actions → `assertOwner(tripId)` → validate → DB write →
`updateTag('trip:'+tripId)`. **Owner-editable item fields:** `name`,
`category`, `scope`, `baseQty`, `unit`, `note` (changing `scope`/`baseQty`
recomputes `requiredQty`). `reorderItem` takes `{ tripId, itemId,
beforeItemId | newIndex }`. (Resolves review G1 / minor reorder-signature.)

### Share & join
`ShareLink` copies `location.href`. A new visitor without a participant token
sees `JoinTripDialog`; `joinTrip(tripId, name)` inserts a `participant`, sets
the participant-token cookie, `updateTag('trip:'+tripId)`.

### Claim items & "still needed"
`claimItem({ tripId, itemId, qty })` / `unclaimItem({ tripId, itemId })` →
`assertParticipant(tripId)` → **`participantId` is resolved server-side from
the `bc_participant` cookie, never passed in the action input** (frozen in
WS-0.8; resolves review I-C) → upsert/delete in `claims` →
`updateTag('trip:'+tripId)`. The page computes, per item:
`needed = requiredQty(item, participantCount)` and
`claimed = Σ claims.qty`; `shortfall = max(0, needed - claimed)`. "Still
needed" lists items with `shortfall > 0`; "Who's bringing what" groups claims
by participant. See `packing-engine.md` for the multiplier math.

### Liveness
`RefreshPoller` calls `router.refresh()` on an interval; because the trip page
is uncached, the re-render reflects others' claims/joins. Manual "Refresh"
button does the same on demand. With Cache Components, recently-visited routes
stay mounted (React `<Activity>`) and client state is preserved across
refresh — so the poller must **not clobber in-progress input** (open
claim-qty field, Join dialog). WS-6.10 follows
`02-guides/preserving-ui-state.md` and tests this (resolves review G3).

## Security notes (per Next.js mutation guidance)

- Every Server Action starts by resolving identity from cookies and asserting
  the caller is the **owner** (edit list / rename) or a **participant** (claim)
  for that exact `tripId`. Direct POSTs without the right token are rejected.
- Trip ids are unguessable (`crypto.randomUUID()`, 122-bit); owner/participant
  tokens are ≥128-bit (`lib/ids.ts` `token()`). The link *is* the capability,
  so links should be treated as secrets by users (documented in UI copy).
  Possession of a link granting join is accepted (D3).
- No secrets reach the client: DB access, `DATABASE_URL`, and the RIDB key
  stay in Server Components / Server Actions / scripts only. `@prisma/client`
  is imported **only** under `lib/db/*`.

## Deployment (Neon + Prisma — D2/DR-6)

Persistence is Neon serverless Postgres, reachable over HTTP/WebSocket via
`@prisma/adapter-neon`, so the SQLite single-node constraint is gone and the
earlier "can't use Vercel" problem is resolved.

| Target | Status | Notes |
|---|---|---|
| Node container / VPS / Fly | ✅ | `next start`; `DATABASE_URL` → Neon |
| **Vercel** | ✅ | Neon branch auto-created per preview deploy |
| Cloudflare (OpenNext) | ✅ driver-compatible | Neon driver runs on workerd; the Next-on-CF adapter must still pass the compat suite (verify) |

- **Build:** `prisma generate` (postinstall + build). **Release:** `prisma
  migrate deploy` as a CI step against `DIRECT_URL` — never at request time.
- App queries use the **pooled** Neon URL; migrations use the **unpooled**
  `DIRECT_URL`.
- Multi-instance is fine (Postgres handles concurrency). Cache Components
  multi-instance consistency still benefits from a shared `cacheHandlers`
  backend per the installed Next.js *deploying-to-platforms* guide —
  unchanged by this decision.
- Local/test: Docker Compose Postgres / ephemeral Testcontainers — see
  `local-dev.md`.
